# 🚀 Task 04 - Init Containers and Sidecar Containers

## 📌 Objective

Learn how Kubernetes handles **Init Containers** and **Sidecar Containers** within a single Pod, and understand the order in which containers are started.

---

## 🎯 Learning Outcomes

By the end of this lab, I can:

- Create a Pod with an Init Container.
- Add multiple containers to a single Pod.
- Understand the difference between Init Containers and Sidecar Containers.
- Verify that Init Containers complete before the application starts.
- Inspect container status and logs using `kubectl`.

---

## 📂 Pod Architecture

```text
                Pod Starts
                     │
                     ▼
          Init Container Starts
                     │
          Performs initialization
                     │
          Completes Successfully
                     ▼
      -----------------------------
      │                           │
      ▼                           ▼
 Main Container            Sidecar Container
     Running                    Running
```

---

## 📝 Pod Configuration

The Pod consists of:

- **Init Container**
  - Runs once during Pod startup.
  - Simulates initialization by sleeping for 5 seconds.
  - Exits successfully before any application containers start.

- **Main Container**
  - Runs continuously.
  - Prints a message every 30 seconds.

- **Sidecar Container**
  - Runs alongside the main container.
  - Prints a monitoring message every 10 seconds.

## 🔍 Verification

### Check Pod Status

```bash
kubectl get pods
```

Expected output:

```text
NAME        READY   STATUS
init-demo   2/2     Running
```

---

### Watch Pod Startup

```bash
kubectl get pods -w
```

Observe the Pod lifecycle:

```text
Init:0/1
↓
PodInitializing
↓
Running (2/2)
```

This confirms that the **Init Container blocks the main containers until it completes**.

---

### Describe the Pod

```bash
kubectl describe pod init-demo
```

Verify:

- Init Container status is **Completed**.
- Main Container status is **Running**.
- Sidecar Container status is **Running**.

---

### View Init Container Logs

```bash
kubectl logs init-demo -c init-container
```

Expected:

```text
Init container running
```

---

### View Main Container Logs

```bash
kubectl logs init-demo -c app
```

Expected:

```text
Main container running
```

---

### View Sidecar Container Logs

```bash
kubectl logs init-demo -c sidecar
```

Expected:

```text
Sidecar container monitoring...
```

---

## 📚 Key Takeaways

- Init Containers always run **before** application containers.
- Main and Sidecar containers start **only after** the Init Container completes.
- Multiple containers inside the same Pod run simultaneously.
- A Pod's container list is **immutable**. To add or remove containers, the Pod must be recreated.
